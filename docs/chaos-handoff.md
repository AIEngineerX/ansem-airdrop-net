## Executive Summary

Ghost wants a simple web dashboard to track Ansem-related airdrops and the main $ANSEM token value.

Core product:

```text
tracked source wallet(s) → outgoing transfers → recipient wallets → optional X reply match
```

The dashboard should answer:

1. How much value has the tracked wallet or wallet cluster sent?
2. How much main $ANSEM has been sent?
3. What is $ANSEM currently worth?
4. Who received transfers?
5. Which recipients can be matched to X replies where they posted Solana addresses?
6. Is the activity from one source wallet or multiple candidate wallets?

Do **not** frame this as a trading signal. It is an on-chain transparency dashboard.

## Accuracy Requirements

### Do Not Count Wallet History Summaries Naively

Helius wallet summaries may show balance changes, but dashboard accounting must parse direction.

For SPL / Token-2022:

```text
source token-account delta < 0
recipient token-account delta > 0
```

For native SOL:

```text
source SOL delta negative
recipient SOL delta positive
exclude transaction fee
```

### Deduplication Key

One transaction can have multiple recipients and multiple assets. Use:

```text
signature + source_wallet + recipient_wallet + mint + amount_raw
```

### Failed Transactions

Exclude failed transactions from totals.

### Batched Airdrops

A single signature may contain many recipient transfers. Split into one row per recipient transfer.

### Main ANSEM Token

Only count main ANSEM when mint equals:

```text
9cRCn9rGT8V2imeM2BaKs13yhMEais3ruM3rPvTGpump
```

Do not rely on symbol alone.

## Source Wallet Discovery

Do not assume Ansem only uses `GV6U...C52`.

Use a `source_wallets` registry with confidence levels.

Confidence levels:

| Level | Meaning |
|---|---|
| confirmed | Ansem publicly posted/signed/officially linked the wallet |
| strong-not-confirmed | public tracker/pump.fun profile + airdrop pattern |
| candidate | funding/flow/path similarity, not verified |
| rejected | investigated and rejected |

Candidate wallet detection:

1. wallets funding primary source wallet;
2. wallets receiving large chunks from source then distributing;
3. wallets with similar airdrop burst behavior;
4. wallets tied to the same pump.fun profile/fee routes;
5. wallets sending to X-reply wallet addresses;
6. wallets distributing main ANSEM to many recipients;
7. wallets repeatedly interacting with source wallet.

UI should show separate totals:

```text
Primary wallet total
Candidate cluster total
Verified-source total
```

Do not merge candidate wallets automatically.

## Database Schema

Use Postgres-compatible schema. SQLite is acceptable locally if easier.

### source_wallets

```sql
CREATE TABLE source_wallets (
  wallet_address TEXT PRIMARY KEY,
  label TEXT NOT NULL,
  confidence TEXT NOT NULL,
  evidence_url TEXT,
  active BOOLEAN NOT NULL DEFAULT TRUE,
  first_seen TIMESTAMPTZ,
  last_seen TIMESTAMPTZ,
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now()
);
```

### transfers

```sql
CREATE TABLE transfers (
  id BIGSERIAL PRIMARY KEY,
  signature TEXT NOT NULL,
  slot BIGINT,
  block_time TIMESTAMPTZ NOT NULL,
  source_wallet TEXT NOT NULL,
  recipient_wallet TEXT NOT NULL,
  mint TEXT NOT NULL,
  symbol TEXT,
  amount_raw NUMERIC NOT NULL,
  amount_ui NUMERIC NOT NULL,
  decimals INT,
  transfer_type TEXT NOT NULL, -- native_sol | spl_token | token_2022
  source_token_account TEXT,
  recipient_token_account TEXT,
  tx_fee_lamports BIGINT,
  parser_confidence TEXT NOT NULL DEFAULT 'medium',
  created_at TIMESTAMPTZ DEFAULT now(),
  event_index INT NOT NULL,
  UNIQUE(signature, source_wallet, recipient_wallet, mint, amount_raw, event_index)
);
```

### token_prices

```sql
CREATE TABLE token_prices (
  id BIGSERIAL PRIMARY KEY,
  mint TEXT NOT NULL,
  symbol TEXT,
  price_usd NUMERIC,
  liquidity_usd NUMERIC,
  market_cap NUMERIC,
  source TEXT NOT NULL,
  captured_at TIMESTAMPTZ NOT NULL
);
```

### x_wallet_mentions

```sql
CREATE TABLE x_wallet_mentions (
  id BIGSERIAL PRIMARY KEY,
  wallet_address TEXT NOT NULL,
  x_handle TEXT,
  tweet_id TEXT NOT NULL,
  tweet_url TEXT,
  parent_tweet_id TEXT,
  reply_text TEXT,
  posted_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ DEFAULT now(),
  UNIQUE(wallet_address, tweet_id)
);
```

### recipient_matches

```sql
CREATE TABLE recipient_matches (
  id BIGSERIAL PRIMARY KEY,
  transfer_id BIGINT REFERENCES transfers(id),
  x_mention_id BIGINT REFERENCES x_wallet_mentions(id),
  match_type TEXT NOT NULL,
  confidence TEXT NOT NULL,
  reason TEXT,
  created_at TIMESTAMPTZ DEFAULT now()
);
```

### unparsed_transactions

```sql
CREATE TABLE unparsed_transactions (
  signature TEXT PRIMARY KEY,
  source_wallet TEXT,
  block_time TIMESTAMPTZ,
  reason TEXT,
  raw_json JSONB,
  created_at TIMESTAMPTZ DEFAULT now()
);
```
